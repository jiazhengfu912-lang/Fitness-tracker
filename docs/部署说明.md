# Vercel 部署说明

## 当前阶段目标

本阶段只为 `Fitness Tracker` 项目完成 Vercel 部署前准备，包括部署说明和环境变量示例文件。

本阶段不修改前端业务代码、不修改数据库 SQL、不写入任何真实密钥。

## 部署平台

- 部署平台：Vercel
- 代码仓库：GitHub
- 当前前端项目目录：`frontend`

## 前置条件

1. GitHub 仓库已经推送完成。
2. Supabase 项目已经创建。
3. `supabase/sql/001_init_schema.sql` 已执行。
4. `supabase/sql/002_add_body_metrics.sql` 已执行。
5. Supabase Auth 可正常注册用户。
6. 业务表已创建并启用 RLS。

## Vercel 导入项目步骤

1. 打开 Vercel 控制台。
2. 点击 `Add New Project`。
3. 选择 GitHub 仓库 `fitness-tracker`。
4. 进入项目导入配置页面。
5. 将 `Root Directory` 设置为 `frontend`。
6. 将 `Framework Preset` 设置为 `Vite`。
7. 确认 `Install Command` 为 `npm install`。
8. 确认 `Build Command` 为 `npm run build`。
9. 确认 `Output Directory` 为 `dist`。
10. 配置环境变量后点击部署。

## Vercel 构建配置

| 配置项 | 值 |
| --- | --- |
| Root Directory | `frontend` |
| Framework Preset | `Vite` |
| Install Command | `npm install` |
| Build Command | `npm run build` |

## 环境变量配置

Vercel 中必须配置以下变量：

1. `VITE_SUPABASE_URL`
2. `VITE_SUPABASE_ANON_KEY`

变量值获取位置：

1. 打开 Supabase 项目后台。
2. 进入 `Project Settings`。
3. 找到 `API Keys`。
4. 复制项目 `Project URL` 作为 `VITE_SUPABASE_URL`。
5. 复制 `publishable key` 或 `anon public key` 作为 `VITE_SUPABASE_ANON_KEY`。

安全要求：

1. 禁止使用 `service_role` key。
2. 禁止使用 `secret key`。
3. 不要把真实密钥提交到 Git。
4. 不要把真实密钥写入文档。
5. 本地开发时真实值放在 `frontend/.env.local`。
6. 线上部署时真实值放在 Vercel `Environment Variables`。

## Supabase 检查项

部署前应确认：

1. `Authentication -> Users` 可注册用户。
2. `Table Editor` 中存在业务表。
3. RLS 已在业务表上开启。
4. 前端使用的是 `anon` 级别公开密钥，而不是高权限密钥。

## 部署后测试路径

部署成功后，至少检查以下页面和流程：

1. `/register`
2. `/login`
3. `/`
4. `/today`
5. `/stats`
6. `/history`
7. `/weight`

建议测试顺序：

1. 打开 `/register`，确认可以注册账号。
2. 打开 `/login`，确认可以登录。
3. 登录后进入 `/`，确认仪表盘加载正常。
4. 进入 `/today`，确认今日记录页正常加载。
5. 进入 `/stats`，确认基础统计页正常加载。
6. 进入 `/history`，确认历史记录页正常加载。
7. 进入 `/weight`，确认体重记录页正常加载。

## 常见错误

### 1. 页面空白

排查顺序：

1. 检查 Vercel 构建是否成功。
2. 检查 `VITE_SUPABASE_URL` 和 `VITE_SUPABASE_ANON_KEY` 是否已配置。
3. 检查前端是否因为缺少环境变量而在启动时抛错。

### 2. Supabase 连接失败

排查顺序：

1. 检查 Supabase `Project URL` 是否填写正确。
2. 检查 `anon public key` 是否填写正确。
3. 检查是否误用了 `service_role` 或 `secret key`。
4. 检查 Supabase 项目是否仍可访问。

### 3. 环境变量没配置

现象：

1. 构建或运行时报缺少 `VITE_SUPABASE_URL`
2. 构建或运行时报缺少 `VITE_SUPABASE_ANON_KEY`

处理方式：

1. 在 Vercel 项目设置中补充环境变量。
2. 重新触发部署。

### 4. Root Directory 选错

现象：

1. Vercel 找不到 Vite 项目入口
2. 安装或构建命令在仓库根目录执行，导致失败

处理方式：

1. 回到 Vercel 项目设置。
2. 将 `Root Directory` 改为 `frontend`。
3. 重新部署。

### 5. Build 失败

排查顺序：

1. 本地先执行 `cd frontend` 和 `npm run build`。
2. 检查 TypeScript 报错。
3. 检查依赖是否缺失。
4. 检查环境变量是否缺失。
5. 检查 `Build Command` 是否设置为 `npm run build`。

## 回滚方式

出现部署问题时，可使用以下方式回滚：

1. 在 `Vercel -> Deployments` 中选择上一个成功版本并恢复。
2. 或在 Git 中执行 `git revert` 生成回滚提交后重新 `push`，让 Vercel 自动部署回滚版本。

## 风险与修复方式

### 风险 1：环境变量配置错误

修复方式：

1. 核对 Supabase `Project URL`。
2. 核对 `anon public key`。
3. 删除错误值后重新部署。

### 风险 2：数据库初始化不完整

修复方式：

1. 确认 `001_init_schema.sql` 已执行。
2. 确认 `002_add_body_metrics.sql` 已执行。
3. 在 Supabase `Table Editor` 中检查表和 RLS 状态。

### 风险 3：部署目录选错

修复方式：

1. 将 Vercel `Root Directory` 改为 `frontend`。
2. 重新部署。
