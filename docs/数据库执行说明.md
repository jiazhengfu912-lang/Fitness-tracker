# 数据库执行说明

## 当前阶段目标

本阶段只初始化 Supabase PostgreSQL 数据库结构和 Row Level Security 访问规则，为后续每日记录 CRUD 页面做准备。

本阶段不写前端页面、不连接远程数据库、不执行 SQL，也不使用任何真实密钥。

## 执行文件

需要执行的 SQL 文件：

```text
supabase/sql/001_init_schema.sql
```

该文件会创建健身记录 MVP 所需业务表、索引、`updated_at` 自动更新时间触发器和 RLS 策略。

## 执行步骤

1. 打开 Supabase 项目后台。
2. 进入 `SQL Editor`。
3. 新建一个 SQL 查询。
4. 打开本仓库中的 `supabase/sql/001_init_schema.sql`。
5. 复制 SQL 文件全部内容。
6. 粘贴到 Supabase `SQL Editor`。
7. 点击执行。
8. 等待执行完成，确认没有错误。

## 账号与密钥要求

1. 用户账号必须使用 Supabase Auth。
2. 数据库脚本不会创建 `users` 表。
3. 数据库脚本不会保存 `password_hash`。
4. 禁止在前端、文档或 SQL 中写入 `service_role` key、secret key 或其他真实密钥。
5. 前端只能使用 `VITE_SUPABASE_URL` 和 `VITE_SUPABASE_ANON_KEY`。

## 如何检查 RLS 是否开启

在 Supabase 后台进入 `Authentication -> Users`，确认当前项目中已有测试用户。业务表的用户 ID 都会关联 Supabase Auth 的 `auth.users.id`。

然后在 `SQL Editor` 中执行以下只读检查：

```sql
select schemaname, tablename, rowsecurity
from pg_tables
where schemaname = 'public'
  and tablename in (
    'profiles',
    'user_preferences',
    'strength_exercises',
    'strength_workout_sessions',
    'strength_workout_exercises',
    'strength_workout_sets',
    'cardio_records',
    'foods',
    'meal_records',
    'meal_items'
  )
order by tablename;
```

预期结果：

1. 返回 10 张表。
2. 每张表的 `rowsecurity` 都是 `true`。

也可以检查策略是否存在：

```sql
select schemaname, tablename, policyname, cmd
from pg_policies
where schemaname = 'public'
  and tablename in (
    'profiles',
    'user_preferences',
    'strength_exercises',
    'strength_workout_sessions',
    'strength_workout_exercises',
    'strength_workout_sets',
    'cardio_records',
    'foods',
    'meal_records',
    'meal_items'
  )
order by tablename, policyname;
```

预期结果是每张表至少有 `select`、`insert`、`update`、`delete` 四类访问策略。

## 访问规则说明

`profiles` 使用以下隔离规则：

```text
id = auth.uid()
```

其他业务表使用以下隔离规则：

```text
user_id = auth.uid()
```

这意味着登录用户只能读取、创建、更新、删除自己的业务数据。前端隐藏按钮或路由保护不能替代 RLS。

## 常见错误处理

### 1. 提示 relation already exists

说明部分表已经存在。当前脚本使用 `create table if not exists`，正常情况下不会因为表存在而失败。

如果你曾经手动创建过字段不一致的同名表，需要先确认是否可以迁移或重建。不要直接删除生产数据表。

### 2. 提示 policy already exists

当前脚本会先删除同名策略再创建同名策略。若仍然报错，检查是否存在手动创建的同名或冲突策略。

### 3. 提示 permission denied for schema auth

确认你是在 Supabase 项目的 `SQL Editor` 中执行，而不是用前端 anon key 执行。

不要把 `service_role` key 写入项目代码或文档。数据库初始化应在 Supabase 控制台中由有权限的项目成员执行。

### 4. 前端读取业务表为空

优先检查：

1. 用户是否已经登录。
2. 当前记录的 `user_id` 是否等于当前登录用户的 `auth.users.id`。
3. RLS 策略是否已创建。
4. 前端是否使用了正确的 `VITE_SUPABASE_URL` 和 `VITE_SUPABASE_ANON_KEY`。

### 5. 插入记录失败

优先检查：

1. 是否传入了 `user_id`。
2. `user_id` 是否等于当前登录用户 ID。
3. 数值是否满足约束，例如 `weight_kg >= 0`、`reps > 0`、`duration_minutes > 0`、`amount_g > 0`。
4. `meal_type`、`weight_unit`、`distance_unit`、`week_start_day` 是否在允许范围内。
