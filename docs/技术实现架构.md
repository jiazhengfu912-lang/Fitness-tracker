# 每日健身记录网站技术实现架构

## 1. 技术栈总览

### 1.1 固定技术栈

| 层级 | 技术 | 用途 |
| --- | --- | --- |
| 前端框架 | React | 构建单页应用界面 |
| 构建工具 | Vite | 本地开发、打包构建 |
| 开发语言 | TypeScript | 提供类型约束，降低长期维护成本 |
| 样式方案 | Tailwind CSS | 快速构建响应式界面 |
| 后端服务 | Supabase | 提供认证、数据库、权限和 API 能力 |
| 认证服务 | Supabase Auth | 管理注册、登录、退出、Session |
| 数据库 | Supabase PostgreSQL | 持久化保存训练、饮食、有氧和用户资料 |
| 权限隔离 | Supabase Row Level Security | 保证用户只能访问自己的数据 |
| 部署平台 | Vercel | 部署前端静态应用 |
| 代码管理 | Git + GitHub | 版本管理、代码托管和部署触发 |

### 1.2 架构定位

本项目采用前端单页应用加 Supabase 后端能力的架构。React 应用直接通过 Supabase 客户端访问 Supabase Auth 和 PostgreSQL 数据表。

核心原则：

1. 不自建账号认证后端。
2. 不设计自建 `users` 密码表。
3. 不在业务数据库保存 `password_hash`。
4. 用户认证统一使用 Supabase Auth 内置的 `auth.users`。
5. 所有业务表必须包含 `user_id` 字段，并关联 `auth.users.id`。
6. 用户数据隔离依赖 Supabase PostgreSQL 的 Row Level Security。
7. 前端登录态通过 Supabase session 管理。
8. 历史记录和长期记忆的本质是 PostgreSQL 持久化存储，不依赖浏览器本地缓存。

### 1.3 不采用的方案

1. 不引入自建 Express 后端。
2. 不引入 Next.js。
3. 不引入 MongoDB。
4. 不引入 Firebase。
5. 不混用 Supabase Auth 和自建用户密码系统。
6. MVP 不引入额外全局状态库，优先使用 React 自身能力和 Supabase session。

### 3.2 路由保护规则

1. 访问需要登录的页面时，如果 Supabase session 不存在，跳转到 `/login`。
2. 已登录用户访问 `/login` 或 `/register` 时，跳转到 `/`。
3. 页面刷新后先恢复 Supabase session，再决定展示页面或跳转。
4. 前端路由保护只用于用户体验，不能替代数据库 RLS。
5. 所有业务数据查询仍必须受 RLS 限制。

## 4. Supabase Auth 认证流程

### 4.1 账号来源

账号认证必须使用 Supabase Auth。Supabase 会在内部维护 `auth.users`，其中包含用户 ID、邮箱、认证状态等认证信息。

本项目不得创建自建 `users` 密码表，也不得保存 `password_hash`。业务侧只创建 `profiles` 表保存用户展示资料和健身相关资料。

### 4.2 注册流程

1. 用户进入 `/register`。
2. 用户填写邮箱、密码和确认密码。
3. 前端做基础校验：邮箱格式、密码长度、两次密码一致。
4. 前端调用 Supabase Auth 注册能力。
5. Supabase Auth 创建 `auth.users` 记录。
6. 注册成功后，系统创建或初始化该用户的 `profiles` 和 `user_preferences` 业务记录。
7. 用户进入首页或登录页，具体取决于 Supabase 项目是否开启邮箱确认。

### 4.3 登录流程

1. 用户进入 `/login`。
2. 用户输入邮箱和密码。
3. 前端调用 Supabase Auth 登录能力。
4. 登录成功后，Supabase 返回 session。
5. 前端保存 Supabase session 的客户端状态。
6. 页面跳转到首页仪表盘。
7. 后续所有业务查询都基于当前 session 对应的用户身份执行。

### 4.4 退出流程

1. 用户点击退出。
2. 前端调用 Supabase Auth 退出能力。
3. Supabase 清除当前 session。
4. 前端清空用户相关状态。
5. 跳转到 `/login`。

### 4.5 Session 管理

1. 前端登录态以 Supabase session 为准。
2. 应用初始化时读取当前 session。
3. 监听 Supabase Auth 状态变化，处理登录、退出、刷新 token 等状态。
4. 不使用 localStorage 保存业务数据作为长期记忆。
5. localStorage 即使被 Supabase SDK 用于 session，也只代表认证状态缓存，不代表业务数据存储。

## 5. Supabase 数据库表设计原则

### 5.1 认证与业务数据分离

| 数据类型 | 存储位置 | 说明 |
| --- | --- | --- |
| 账号 ID、邮箱、认证状态 | Supabase Auth 的 `auth.users` | 由 Supabase 管理 |
| 密码与认证凭据 | Supabase Auth 内部 | 业务表不可保存 |
| 用户昵称、身高、体重等资料 | `profiles` | 通过 `user_id` 关联 `auth.users.id` |
| 力量训练、有氧、饮食记录 | 业务表 | 每张表都包含 `user_id` |
| 统计结果 | 默认实时查询计算 | MVP 不单独落统计表 |

### 5.2 user_id 设计原则

1. 每张业务表都必须包含 `user_id`。
2. `user_id` 类型与 `auth.users.id` 保持一致。
3. `user_id` 必须关联 `auth.users.id`。
4. 前端新增业务数据时，`user_id` 必须来自当前 Supabase session 对应的用户 ID。
5. RLS 必须校验当前登录用户只能访问 `user_id` 等于自己的记录。
6. 不允许前端通过传入其他用户 ID 获取或修改他人数据。

### 5.3 主键与时间字段

所有业务表建议包含：

| 字段 | 说明 |
| --- | --- |
| `id` | 表主键，建议使用 UUID |
| `user_id` | 当前记录所属用户，关联 `auth.users.id` |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 5.4 长期记忆说明

本产品的长期记忆不是前端缓存，也不是浏览器本地文件，而是 Supabase PostgreSQL 中的持久化业务表。

用户每次创建、编辑、删除训练、有氧或饮食记录时，最终都必须写入 PostgreSQL。用户退出登录、关闭浏览器或更换设备后，只要重新登录同一个 Supabase Auth 账号，就可以通过 PostgreSQL 查询到历史数据。

3. `user_id` 与所属餐次记录的 `user_id` 必须一致。

### 6.11 user_preferences

| 字段 | 类型说明 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | UUID | 是 | 偏好记录 ID |
| `user_id` | UUID | 是 | 关联 `auth.users.id`，唯一 |
| `weight_unit` | 字符串 | 是 | 默认 kg |
| `distance_unit` | 字符串 | 是 | 默认 km |
| `week_start_day` | 字符串 | 是 | 默认 Monday |
| `created_at` | 时间 | 是 | 创建时间 |
| `updated_at` | 时间 | 是 | 更新时间 |

设计说明：

1. MVP 默认使用 kg、km、Monday。
2. 即使设置页延后，该表也可在注册后初始化，方便后续扩展。

### 6.12 表之间的关系

| 关系 | 说明 |
| --- | --- |
| `auth.users` 1 对 1 `profiles` | 每个认证用户最多一条业务资料 |
| `auth.users` 1 对 1 `user_preferences` | 每个认证用户最多一条偏好记录 |
| `auth.users` 1 对多 `strength_exercises` | 每个用户拥有自己的动作库 |
| `auth.users` 1 对多 `strength_workout_sessions` | 每个用户有多次力量训练会话 |
| `strength_workout_sessions` 1 对多 `strength_workout_exercises` | 一次训练包含多个动作 |
| `strength_workout_exercises` 1 对多 `strength_workout_sets` | 一个动作包含多组 |
| `strength_exercises` 1 对多 `strength_workout_exercises` | 一个动作可被多次训练引用 |
| `auth.users` 1 对多 `cardio_records` | 每个用户有多条有氧记录 |
| `auth.users` 1 对多 `foods` | 每个用户拥有自己的食物库 |
| `auth.users` 1 对多 `meal_records` | 每个用户有多条餐次记录 |
| `meal_records` 1 对多 `meal_items` | 一个餐次包含多个食物明细 |
| `foods` 1 对多 `meal_items` | 一个食物可被多次饮食记录引用 |

## 7. RLS 权限策略设计

### 7.1 RLS 定位

Row Level Security 是本项目数据隔离的核心。前端页面隐藏、路由守卫、表单校验都不能替代 RLS。

所有业务表都必须开启 RLS。开启后，除非配置明确策略，否则客户端不能随意读取或修改表数据。

### 7.2 通用策略口径

每张业务表至少需要以下策略口径：

| 操作 | 策略口径 |
| --- | --- |
| 查询 | 当前登录用户只能查询 `user_id` 等于自己用户 ID 的记录 |
| 新增 | 当前登录用户只能新增 `user_id` 等于自己用户 ID 的记录 |
| 更新 | 当前登录用户只能更新 `user_id` 等于自己用户 ID 的记录，并且更新后仍归属于自己 |
| 删除 | 当前登录用户只能删除 `user_id` 等于自己用户 ID 的记录 |

### 7.3 各表 RLS 要点

| 表名 | RLS 要点 |
| --- | --- |
| `profiles` | 只能读取、创建、更新、删除自己的资料 |
| `user_preferences` | 只能读取、创建、更新、删除自己的偏好 |
| `strength_exercises` | 只能管理自己的动作 |
| `strength_workout_sessions` | 只能管理自己的训练会话 |
| `strength_workout_exercises` | 只能管理自己的训练动作记录 |
| `strength_workout_sets` | 只能管理自己的训练组 |
| `cardio_records` | 只能管理自己的有氧记录 |
| `foods` | 只能管理自己的食物 |
| `meal_records` | 只能管理自己的餐次 |
| `meal_items` | 只能管理自己的食物明细 |

### 7.4 关联表一致性要求

仅有单表 RLS 不足以表达所有业务一致性，开发时还需要保证：

1. 新增 `strength_workout_exercises` 时，所属 `session_id` 必须属于当前用户。
2. 新增 `strength_workout_sets` 时，所属 `workout_exercise_id` 必须属于当前用户。
3. 新增 `meal_items` 时，所属 `meal_record_id` 必须属于当前用户。
4. 新增 `meal_items` 时，如传入 `food_id`，该食物必须属于当前用户。
5. 新增 `strength_workout_exercises` 时，如传入 `exercise_id`，该动作必须属于当前用户。
6. 前端表单不能依赖用户手动输入 `user_id`，应从当前 session 派生。

### 7.5 RLS 验证用例

MVP 完成后必须验证：

1. 用户 A 创建力量训练记录，用户 B 查询不到。
2. 用户 A 创建有氧记录，用户 B 查询不到。
3. 用户 A 创建饮食记录，用户 B 查询不到。
4. 用户 B 不能通过修改请求参数更新用户 A 的记录。
5. 用户 B 不能通过修改请求参数删除用户 A 的记录。
6. 未登录状态不能读取业务表数据。
7. 未登录状态不能写入业务表数据。

## 8. 前端状态管理设计

### 8.1 状态分类

| 状态类型 | 管理方式 | 说明 |
| --- | --- | --- |
| 认证状态 | Supabase session + React Context | 当前用户、登录中、退出 |
| 页面筛选状态 | 页面组件本地状态 | 日期、日期范围、选中的 tab |
| 表单编辑状态 | 页面或表单组件本地状态 | 当前正在填写的训练、饮食、有氧数据 |
| 远程业务数据 | 自定义 hooks 调用 Supabase | 从 PostgreSQL 读取 |
| 全局偏好 | 登录后读取 `user_preferences` | 单位、周起始日 |
| 临时提示 | 局部状态 | 保存成功、失败、加载中 |

### 8.2 MVP 状态管理原则

1. 不引入 Redux、Zustand 等额外状态库。
2. 认证状态通过 `AuthProvider` 类似结构集中提供。
3. 业务数据不要长期复制到全局状态中。
4. 页面进入时按需加载数据，保存成功后重新拉取或局部更新。
5. 表单未保存数据只保留在当前页面状态中。
6. 页面刷新后，业务数据应从 Supabase PostgreSQL 重新加载。

### 8.3 错误与加载状态

每个数据读写流程都需要明确：

1. 初始加载中。
2. 空数据状态。
3. 保存中。
4. 保存成功。
5. 保存失败。
6. 删除确认。
7. 删除失败。
8. 登录失效或 session 过期。

## 9. 数据读写流程

### 9.1 通用读流程

1. 页面加载。
2. 获取当前 Supabase session。
3. 如果没有 session，跳转登录页。
4. 根据页面需要发起 Supabase 查询。
5. 查询条件包含业务日期、日期范围或记录 ID。
6. RLS 自动限制只能返回当前用户的数据。
7. 前端将数据映射成页面展示结构。

### 9.2 通用写流程

1. 用户填写表单。
2. 前端做基础校验。
3. 从当前 session 获取用户 ID。
4. 组装业务数据，写入对应表。
5. 数据中的 `user_id` 使用当前 session 用户 ID。
6. Supabase 执行写入。
7. RLS 校验该用户是否允许写入。
8. 成功后刷新当前页面数据或局部更新。
9. 失败时展示错误提示。

### 9.3 力量训练保存流程

1. 用户选择日期。
2. 用户添加训练动作。
3. 对新动作名称，先确认是否已有当前用户的 `strength_exercises`。
4. 不存在时创建当前用户自己的动作记录。
5. 创建或获取当天 `strength_workout_sessions`。
6. 为会话创建 `strength_workout_exercises`。
7. 为每个动作创建多条 `strength_workout_sets`。
8. 保存完成后重新读取该日期的力量训练数据。

### 9.4 有氧记录保存流程

1. 用户选择日期。
2. 用户填写有氧类型和时长。
3. 前端校验时长大于 0。
4. 写入 `cardio_records`。
5. 保存完成后刷新当天记录和统计摘要。

### 9.5 饮食记录保存流程

1. 用户选择日期和餐次。
2. 用户填写食物名称和摄入重量。
3. 对新食物名称，先确认是否已有当前用户的 `foods`。
4. 不存在时创建当前用户自己的食物记录。
5. 创建或获取对应日期和餐次的 `meal_records`。
6. 写入 `meal_items`，同时保存食物名称快照和可选营养快照。
7. 保存完成后刷新当天饮食记录和统计摘要。

### 9.6 历史记录读取流程

1. 用户进入 `/history`。
2. 默认日期范围为最近 7 天或最近 30 天。
3. 分别读取力量训练、有氧和饮食记录。
4. 按日期聚合成每日摘要。
5. 用户点击某一天进入 `/records/:date` 查看详情。

### 9.7 统计读取流程

1. 用户进入 `/stats`。
2. 默认选择最近 7 天。
3. 根据日期范围读取原始记录。
4. 前端或数据库查询层计算基础统计。
5. 展示训练天数、总组数、总容量、有氧总时长、饮食条目数、摄入重量等。

MVP 中统计结果可以实时计算，不需要单独创建统计表。

## 11. 环境变量设计

### 11.1 前端环境变量

| 变量名 | 必填 | 说明 |
| --- | --- | --- |
| `VITE_SUPABASE_URL` | 是 | Supabase 项目 URL |
| `VITE_SUPABASE_ANON_KEY` | 是 | Supabase anon public key |

### 11.2 安全规则

1. 前端只能使用 Supabase anon public key。
2. 不允许把 Supabase service role key 放进前端项目。
3. 不允许把 service role key 配置到 Vercel 的前端公开环境变量中。
4. `.env.local` 只用于本地开发，不提交到 GitHub。
5. Vercel 中需要配置同名环境变量。
6. 数据权限必须依赖 RLS，而不是依赖隐藏前端 key。

### 11.3 环境划分

| 环境 | 用途 | 说明 |
| --- | --- | --- |
| 本地开发环境 | 日常开发调试 | 使用 `.env.local` |
| 预览环境 | GitHub 分支或 PR 预览 | 使用 Vercel Preview 环境变量 |
| 生产环境 | 正式访问 | 使用 Vercel Production 环境变量 |
