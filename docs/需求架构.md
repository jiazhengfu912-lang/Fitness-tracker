# 每日健身记录网站需求架构

## 1. 产品定位

每日健身记录网站是一个面向个人长期使用的健身与饮食记录系统。用户通过账号登录后，可以持续记录每天的力量训练、有氧运动和饮食摄入，并基于历史数据查看训练进步、运动习惯和饮食趋势。

本产品第一阶段以“稳定记录、长期保存、清晰回看、基础统计”为核心，不追求社交、复杂课程系统或自动化推荐。

## 2. 产品目标

1. 支持用户创建账号、登录和管理个人数据。
2. 支持按日期记录力量训练，包括动作、组数、每组重量和次数。
3. 支持按日期记录有氧训练，包括类型、时长、距离、消耗等信息。
4. 支持按日期记录饮食，包括食物名称、摄入重量和基础营养信息。
5. 将所有训练和饮食数据长期保存到数据库。
6. 支持用户再次登录后查看、编辑、统计历史记录。
7. 提供 MVP 第一版可落地的功能边界，避免范围失控。

## 3. 用户角色

| 角色 | 描述 | 权限范围 |
| --- | --- | --- |
| 未登录用户 | 尚未登录或注册的访问者 | 只能访问登录、注册页面 |
| 已登录用户 | 完成账号登录的个人用户 | 可以创建、查看、编辑、删除自己的训练和饮食记录，查看自己的统计数据 |
| 系统管理员 | 后续可选角色，MVP 不强制实现 | 管理用户状态、系统字典、异常数据，不查看用户隐私数据明细除非有明确授权 |

## 4. 网站核心模块

### 4.1 用户账号模块

#### 模块目标

确保每个用户拥有独立账号和独立数据空间，用户下次登录后可以继续查看和管理自己的长期健身记录。

#### 功能范围

| 功能 | 说明 | MVP 是否必须 |
| --- | --- | --- |
| 用户注册 | 用户使用邮箱和密码创建账号 | 是 |
| 用户登录 | 用户使用邮箱和密码登录 | 是 |
| 用户退出登录 | 清除当前登录状态 | 是 |
| 登录状态保持 | 刷新页面或短时间关闭网站后仍能保持登录 | 是 |
| 密码加密保存 | 数据库只保存密码哈希，不保存明文密码 | 是 |
| 用户基础信息 | 昵称、头像、身高、体重目标等 | 否 |
| 找回密码 | 通过邮箱重置密码 | 否 |
| 第三方登录 | 使用外部账号登录 | 否 |

#### 关键规则

1. 用户只能访问自己的记录。
2. 所有训练、饮食、统计数据都必须关联到用户 ID。
3. 删除账号属于高风险操作，MVP 可以暂不开放。
4. 邮箱必须唯一。
5. 密码必须满足最低安全要求，例如长度不少于 8 位。

### 4.2 力量训练记录模块

#### 模块目标

让用户精确记录每天做了哪些力量训练动作，每个动作做了多少组，每组使用的重量和完成次数。

#### 功能范围

| 功能 | 说明 | MVP 是否必须 |
| --- | --- | --- |
| 新建训练日记录 | 用户选择日期后创建当天力量训练记录 | 是 |
| 添加训练动作 | 在某一天添加一个或多个训练动作 | 是 |
| 记录训练组 | 每个动作下记录多组数据 | 是 |
| 记录每组重量 | 单位默认 kg | 是 |
| 记录每组次数 | 每组完成次数 | 是 |
| 编辑训练记录 | 修改动作、重量、次数、备注 | 是 |
| 删除训练记录 | 删除某天、某动作或某组记录 | 是 |
| 动作名称复用 | 用户可以从历史动作中选择 | 是 |
| 训练备注 | 记录状态、疲劳、动作感受 | 否 |
| RPE/RIR | 记录主观强度或剩余次数 | 否 |
| 训练模板 | 将常用动作组合保存为模板 | 否 |
| 个人最佳记录 | 自动计算最大重量、最大容量等 | 否 |

#### 数据结构层级

1. 训练日期：例如 2026-06-15。
2. 训练会话：某天的一次力量训练。
3. 训练动作：例如深蹲、卧推、硬拉。
4. 训练组：同一动作下的第 1 组、第 2 组、第 3 组。
5. 每组数据：重量、次数、是否热身组、备注。

#### 关键规则

1. 同一天可以有多次力量训练会话，但 MVP 可以默认一天一个力量训练会话。
2. 每个动作必须至少有一组训练数据才算有效记录。
3. 重量允许为 0，用于记录自重训练。
4. 次数必须为正整数。
5. 删除训练会话时，需要同时删除该会话下的动作记录和组记录。

### 4.3 有氧记录模块

#### 模块目标

让用户记录每天完成的有氧运动类型、持续时间和可选指标，用于长期查看运动频率和有氧总量。

#### 功能范围

| 功能 | 说明 | MVP 是否必须 |
| --- | --- | --- |
| 新建有氧记录 | 按日期添加一条有氧运动 | 是 |
| 选择有氧类型 | 跑步、骑行、游泳、椭圆机、跳绳、爬楼等 | 是 |
| 记录运动时长 | 单位默认分钟 | 是 |
| 记录距离 | 跑步、骑行等项目可填写，单位默认 km | 否 |
| 记录消耗热量 | 用户手动填写 | 否 |
| 记录强度 | 低、中、高或自定义主观强度 | 否 |
| 编辑有氧记录 | 修改类型、时长、距离等 | 是 |
| 删除有氧记录 | 删除错误记录 | 是 |

#### 关键规则

1. 一天可以记录多条有氧运动。
2. 有氧时长必须大于 0。
3. 距离和热量为可选字段。
4. 有氧类型优先使用预设选项，同时允许用户输入自定义类型。

### 4.4 饮食记录模块

#### 模块目标

让用户记录每天吃了什么食物、吃了多少克，并为后续热量、蛋白质、碳水、脂肪等统计留出数据结构。

#### 功能范围

| 功能 | 说明 | MVP 是否必须 |
| --- | --- | --- |
| 新建饮食记录 | 按日期添加饮食记录 | 是 |
| 按餐次记录 | 早餐、午餐、晚餐、加餐 | 是 |
| 添加食物 | 输入食物名称 | 是 |
| 记录摄入重量 | 单位默认 g | 是 |
| 编辑食物记录 | 修改食物、重量、备注 | 是 |
| 删除食物记录 | 删除错误记录 | 是 |
| 食物历史复用 | 从用户历史输入的食物中快速选择 | 是 |
| 记录热量 | 每 100g 热量或本次总热量 | 否 |
| 记录三大营养素 | 蛋白质、碳水、脂肪 | 否 |
| 食物库 | 内置常见食物营养数据 | 否 |
| 条码扫描 | 扫描包装食品条码 | 否 |

#### 关键规则

1. 食物名称和摄入重量是 MVP 必填项。
2. 摄入重量必须大于 0。
3. 营养数据在 MVP 可作为可选字段，不阻塞记录流程。
4. 食物名称需要保存快照，避免后续修改食物库名称影响历史记录。

### 4.5 长期数据存储模块

#### 模块目标

保证用户历史训练和饮食数据可长期保存、稳定查询、可追溯、可统计。

#### 功能范围

| 功能 | 说明 | MVP 是否必须 |
| --- | --- | --- |
| 用户数据持久化 | 所有记录保存到数据库 | 是 |
| 按用户隔离数据 | 所有业务表必须包含用户归属关系 | 是 |
| 历史记录查询 | 用户可按日期范围查看历史数据 | 是 |
| 数据编辑留存 | 更新记录时保存更新时间 | 是 |
| 软删除能力 | 删除记录时可标记删除，便于恢复和排查 | 否 |
| 数据备份 | 定期备份数据库 | 否 |
| 数据导出 | 用户导出自己的训练和饮食数据 | 否 |

#### 数据保存原则

1. 账号、训练、饮食、有氧、统计所需原始数据必须入库。
2. 统计结果可以实时计算，MVP 不强制单独落表。
3. 历史记录应尽量保存用户当时填写的文本快照。
4. 所有核心表都应包含创建时间和更新时间。
5. 涉及用户隐私的数据不得跨用户展示。

### 4.6 数据统计模块

#### 模块目标

基于长期记录的数据，帮助用户了解训练频率、训练容量、有氧总量和饮食摄入趋势。

#### MVP 统计范围

| 统计项 | 说明 | MVP 是否必须 |
| --- | --- | --- |
| 每日记录汇总 | 展示某天力量、有氧、饮食是否有记录 | 是 |
| 周训练次数 | 当前周完成力量训练的天数或次数 | 是 |
| 周有氧总时长 | 当前周有氧分钟数合计 | 是 |
| 力量训练总组数 | 指定日期范围内完成的总组数 | 是 |
| 力量训练总容量 | 重量 × 次数 的合计，按动作或日期统计 | 是 |
| 饮食摄入重量 | 每日或每周食物克数合计 | 是 |
| 动作历史趋势 | 某个动作的重量、次数、容量趋势 | 否 |
| 热量和宏量营养趋势 | 热量、蛋白质、碳水、脂肪趋势 | 否 |
| 个人最佳记录 | 单次最大重量、最大组容量、最大总容量 | 否 |

#### 统计页面规则

1. 所有统计默认只计算当前登录用户的数据。
2. 日期范围至少支持最近 7 天、最近 30 天和自定义起止日期。
3. 统计结果必须能追溯到原始记录。
4. MVP 可以使用简单列表和数字卡片，不强制复杂图表。

## 5. 核心业务流程

### 5.1 注册登录流程

1. 用户进入网站。
2. 未登录用户跳转到登录页。
3. 新用户选择注册，填写邮箱和密码。
4. 系统创建账号并保存密码哈希。
5. 用户登录成功后进入首页仪表盘。
6. 系统基于用户 ID 加载该用户自己的历史记录和统计数据。

### 5.2 每日力量训练记录流程

1. 用户进入“今日记录”或“训练记录”页面。
2. 选择记录日期，默认今天。
3. 添加力量训练动作。
4. 在动作下添加训练组。
5. 填写每组重量和次数。
6. 保存记录。
7. 系统写入训练会话、动作记录和训练组数据。
8. 用户可以在历史页面按日期查看或编辑。

### 5.3 每日有氧记录流程

1. 用户进入“今日记录”页面。
2. 添加有氧运动。
3. 选择或输入有氧类型。
4. 填写运动时长。
5. 可选填写距离、热量、强度和备注。
6. 保存记录。
7. 系统写入有氧记录数据。

### 5.4 每日饮食记录流程

1. 用户进入“今日记录”页面。
2. 选择餐次。
3. 添加食物名称。
4. 填写摄入重量。
5. 可选填写热量和营养素。
6. 保存记录。
7. 系统写入餐次记录和食物明细。

### 5.5 历史查看与统计流程

1. 用户进入“历史记录”页面。
2. 选择日期或日期范围。
3. 系统展示该范围内的力量、有氧和饮食记录。
4. 用户进入“统计”页面。
5. 选择最近 7 天、最近 30 天或自定义范围。
6. 系统基于原始记录计算训练次数、训练容量、有氧时长和饮食摄入。

## 6. 页面结构

### 6.1 页面总览

| 页面 | 路径含义 | 访问权限 | MVP 是否必须 |
| --- | --- | --- | --- |
| 登录页 | 用户登录入口 | 未登录用户 | 是 |
| 注册页 | 用户注册入口 | 未登录用户 | 是 |
| 首页仪表盘 | 今日概览和近期统计 | 已登录用户 | 是 |
| 今日记录页 | 记录当天训练、有氧、饮食 | 已登录用户 | 是 |
| 历史记录页 | 按日期查看历史数据 | 已登录用户 | 是 |
| 力量训练详情页 | 查看和编辑某天力量训练 | 已登录用户 | 是 |
| 有氧记录详情页 | 查看和编辑某天有氧记录 | 已登录用户 | 可合并到今日记录页 |
| 饮食记录详情页 | 查看和编辑某天饮食记录 | 已登录用户 | 可合并到今日记录页 |
| 统计页 | 查看训练和饮食统计 | 已登录用户 | 是 |
| 个人设置页 | 管理基础资料和偏好 | 已登录用户 | 否 |

### 6.2 登录页

#### 页面内容

1. 邮箱输入框。
2. 密码输入框。
3. 登录按钮。
4. 跳转注册入口。
5. 登录失败提示。

#### 页面规则

1. 已登录用户访问登录页时，自动进入首页仪表盘。
2. 登录失败时不能暴露账号是否存在的敏感细节。
3. 表单提交前需要校验邮箱格式和密码非空。

### 6.3 注册页

#### 页面内容

1. 邮箱输入框。
2. 密码输入框。
3. 确认密码输入框。
4. 注册按钮。
5. 跳转登录入口。

#### 页面规则

1. 邮箱必须唯一。
2. 两次密码必须一致。
3. 注册成功后可以直接进入首页，或跳转登录页。

### 6.4 首页仪表盘

#### 页面内容

1. 今日日期。
2. 今日力量训练摘要：动作数、总组数、总容量。
3. 今日有氧摘要：有氧次数、总时长。
4. 今日饮食摘要：食物条目数、摄入重量，可选热量。
5. 最近 7 天训练次数。
6. 最近 7 天有氧总时长。
7. 快速入口：记录力量、记录有氧、记录饮食、查看统计。

#### 页面规则

1. 首页只展示摘要，不承担复杂编辑。
2. 摘要数据必须来自数据库中的原始记录。
3. 没有数据时展示空状态，引导用户开始记录。

### 6.5 今日记录页

#### 页面内容

1. 日期选择器，默认今天。
2. 力量训练记录区。
3. 有氧记录区。
4. 饮食记录区。
5. 保存状态提示。

#### 力量训练记录区

1. 添加动作按钮。
2. 动作名称输入或历史选择。
3. 每个动作下展示组列表。
4. 每组包含重量、次数、删除按钮。
5. 添加组按钮。
6. 动作备注。

#### 有氧记录区

1. 添加有氧按钮。
2. 有氧类型。
3. 时长。
4. 距离、热量、强度、备注等可选字段。

#### 饮食记录区

1. 餐次选择。
2. 添加食物按钮。
3. 食物名称。
4. 摄入重量。
5. 可选热量、蛋白质、碳水、脂肪。

### 6.6 历史记录页

#### 页面内容

1. 日历或日期列表。
2. 日期范围筛选。
3. 每日摘要列表。
4. 进入某一天详情的入口。

#### 每日摘要字段

1. 日期。
2. 力量训练：动作数、总组数、总容量。
3. 有氧：记录数、总时长。
4. 饮食：食物条目数、摄入重量。

### 6.7 统计页

#### 页面内容

1. 日期范围筛选。
2. 训练频率统计。
3. 力量训练容量统计。
4. 有氧总时长统计。
5. 饮食摄入统计。
6. 动作维度筛选，MVP 可选。

#### MVP 必须展示的统计指标

1. 日期范围内力量训练天数。
2. 日期范围内力量训练总组数。
3. 日期范围内力量训练总容量。
4. 日期范围内有氧总次数。
5. 日期范围内有氧总时长。
6. 日期范围内饮食记录条目数。
7. 日期范围内饮食摄入总重量。

### 6.8 个人设置页

#### 页面内容

1. 昵称。
2. 身高。
3. 当前体重。
4. 目标体重。
5. 重量单位偏好。
6. 修改密码入口。

#### MVP 范围

个人设置页不是第一版必需页面，可以在后续版本实现。

## 7. 数据库表结构设计

### 7.1 设计原则

1. 所有用户业务数据必须通过 `user_id` 关联到用户。
2. 表结构优先支持长期记录和统计查询。
3. 训练组、食物明细等高频数据单独建表，避免把复杂列表塞进单个字段。
4. 历史记录中对动作名、食物名保留快照，避免基础字典变化影响历史展示。
5. 所有核心表包含 `created_at` 和 `updated_at`。
6. MVP 不绑定具体数据库产品或后端框架，以下为逻辑表结构。

### 7.2 用户表：users

| 字段 | 类型说明 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | 唯一标识 | 是 | 用户 ID |
| email | 字符串 | 是 | 登录邮箱，唯一 |
| password_hash | 字符串 | 是 | 密码哈希 |
| nickname | 字符串 | 否 | 用户昵称 |

## 8. 数据统计口径

### 8.1 力量训练统计

| 指标 | 计算方式 | 说明 |
| --- | --- | --- |
| 训练天数 | 日期范围内存在力量训练会话的去重日期数 | 同一天多次训练仍计为 1 天 |
| 训练次数 | 日期范围内力量训练会话数量 | 同一天多次训练按多次计算 |
| 总组数 | 日期范围内训练组数量 | 默认包含正式组和热身组，后续可筛选 |
| 总次数 | 所有训练组 reps 合计 | 用于观察总重复次数 |
| 总容量 | 所有训练组 weight_kg × reps 合计 | 自重训练重量为 0 时不贡献容量 |
| 单动作容量 | 指定动作下 weight_kg × reps 合计 | 用于动作趋势 |

### 8.2 有氧统计

| 指标 | 计算方式 | 说明 |
| --- | --- | --- |
| 有氧次数 | 日期范围内 cardio_records 数量 | 一条记录算一次 |
| 有氧总时长 | duration_minutes 合计 | 单位分钟 |
| 有氧总距离 | distance_km 合计 | 只统计填写距离的记录 |
| 有氧总消耗 | calories_burned 合计 | 只统计填写热量的记录 |

### 8.3 饮食统计

| 指标 | 计算方式 | 说明 |
| --- | --- | --- |
| 饮食记录天数 | 日期范围内存在 meal_items 的去重日期数 | 至少一条食物记录算一天 |
| 食物条目数 | meal_items 数量 | 每个食物明细算一条 |
| 摄入总重量 | amount_g 合计 | 单位 g |
| 总热量 | calories_snapshot 合计 | 只统计填写热量的记录 |
| 总蛋白质 | protein_g_snapshot 合计 | 后续版本重点指标 |
| 总碳水 | carbs_g_snapshot 合计 | 后续版本重点指标 |
| 总脂肪 | fat_g_snapshot 合计 | 后续版本重点指标 |

## 9. MVP 第一版功能范围

### 9.1 第一版必须实现

1. 用户注册。
2. 用户登录。
3. 用户退出登录。
4. 登录状态保持。
5. 用户只能查看和管理自己的数据。
6. 创建、查看、编辑、删除力量训练记录。
7. 力量训练记录必须支持动作、组、每组重量、每组次数。
8. 创建、查看、编辑、删除有氧记录。
9. 有氧记录必须支持类型和时长。
10. 创建、查看、编辑、删除饮食记录。
11. 饮食记录必须支持餐次、食物名称、摄入重量。
12. 所有账号、训练、有氧、饮食数据必须保存到数据库。
13. 用户重新登录后可以查看历史数据。
14. 首页展示今日记录摘要。
15. 历史记录页支持按日期查看。
16. 统计页展示最近 7 天和最近 30 天基础统计。
17. 数据库表至少覆盖 users、strength_workout_sessions、strength_workout_exercises、strength_workout_sets、cardio_records、meal_records、meal_items。

### 9.2 第一版不做

1. 社交动态、关注、点赞、评论。
2. 课程售卖或教练管理。
3. AI 自动生成训练计划。
4. 自动识别食物图片。
5. 条码扫描。
6. 第三方登录。
7. 手机 App。
8. 复杂图表和高级数据分析。
9. 可穿戴设备同步。
10. 多人家庭或团队账号。
11. 支付订阅。

### 9.3 MVP 验收标准

1. 新用户可以注册并登录。
2. 登录后创建的力量训练记录会保存到数据库。
3. 力量训练可以记录多个动作，每个动作可以记录多组重量和次数。
4. 登录后创建的有氧记录会保存到数据库。
5. 有氧记录至少包含类型和时长。
6. 登录后创建的饮食记录会保存到数据库。
7. 饮食记录至少包含餐次、食物名称和摄入重量。
8. 用户退出并重新登录后，仍能看到之前保存的记录。
9. 历史记录页能按日期查看训练、有氧、饮食数据。
10. 统计页能显示最近 7 天和最近 30 天的基础统计。
11. 一个用户不能看到另一个用户的数据。
12. 删除记录后，页面和统计结果同步更新。

## 10. 后续版本扩展功能

### 10.1 训练增强

1. 训练模板：保存常用训练计划，一键套用到当天。
2. 动作库管理：按部位、器械、动作类型筛选。
3. 个人最佳记录：最大重量、最大次数、最大容量。
4. 训练趋势图：按动作查看重量、次数、容量变化。
5. RPE/RIR 记录：评估训练强度和疲劳。
6. 休息计时器：组间休息提醒。
7. 训练计划周期：按周或月安排训练。

### 10.2 有氧增强

1. 配速统计：跑步按距离和时长计算配速。
2. 心率区间：记录平均心率、最大心率。
3. 有氧类型细分：室内跑、户外跑、公路骑行、动感单车等。
4. 可穿戴设备导入：从运动设备同步有氧记录。

### 10.3 饮食增强

1. 常见食物库：内置营养数据。
2. 自定义食物营养：用户维护每 100g 营养值。
3. 每日热量目标：设置目标并显示达成情况。
4. 蛋白质、碳水、脂肪目标。
5. 食物组合或常用餐：一键添加固定餐食。
6. 条码扫描和包装食品识别。
7. 图片识别食物，仅作为辅助，不直接替代用户确认。

### 10.4 数据与体验增强

1. 数据导出：导出训练和饮食数据。
2. 数据导入：从表格或其他工具迁移历史数据。
3. 高级筛选：按动作、部位、有氧类型、餐次筛选。
4. 图表分析：折线图、柱状图、日历热力图。
5. 目标追踪：体重目标、训练频率目标、蛋白质目标。
6. 提醒功能：未记录提醒、训练计划提醒。
7. 移动端适配优化。
8. 深色模式。

### 10.5 账号与安全增强

1. 邮箱验证。
2. 找回密码。
3. 修改邮箱。
4. 修改密码。
5. 登录设备管理。
6. 数据删除申请。
7. 隐私设置。

## 11. 权限与数据安全要求

1. 所有需要访问业务数据的页面都必须要求登录。
2. 后端查询必须基于当前登录用户 ID 过滤数据。
3. 前端隐藏入口不能替代后端权限校验。
4. 密码必须哈希保存，禁止明文保存。
5. 用户输入内容需要做基础校验，避免空值、非法数值和异常长度。
6. 用户删除记录后，统计结果必须重新计算或同步更新。
7. 生产环境应具备数据库备份机制，MVP 本地开发阶段可先保留设计要求。

## 12. 非功能需求

### 12.1 可用性

1. 记录流程要尽量短，用户打开今日记录页后可以快速添加数据。
2. 重复录入动作和食物时，应支持历史项选择。
3. 没有记录时需要展示明确空状态。
4. 保存成功、失败、删除成功都需要有清晰反馈。

### 12.2 性能

1. 首页和今日记录页应优先加载最近数据。
2. 历史记录页按日期范围查询，避免一次加载全部历史。
3. 统计页默认查询最近 7 天或 30 天。
4. 长期使用后，训练组和饮食明细表会增长较快，需要为用户 ID 和日期字段建立查询优化设计。

### 12.3 可维护性

1. 力量、有氧、饮食模块边界清晰，避免互相耦合。
2. 统计逻辑应集中管理，确保首页、历史页、统计页口径一致。
3. 表字段命名保持统一，例如日期字段使用业务日期，时间字段使用创建和更新时间。
4. MVP 先保证核心记录闭环，再扩展高级分析。
